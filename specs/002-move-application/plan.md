# Implementation Plan: Move Application — Flips and Encirclement

**Branch**: `002-move-application` | **Date**: 2026-05-21 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/002-move-application/spec.md`

## Summary

Implement `applyMove(state, move)` as a pure function that accepts an immutable `GameState` and a `Move`, validates legality, and returns a new `GameState` reflecting the move's effects. Simple moves (PLACE, simple JOIN, PASS) are mechanical state updates. Cyclic JOINs require cycle path extraction (BFS/DFS on the planar graph) and interior-coin detection via a point-in-polygon test. Exit criteria: property tests assert invariant preservation across random move sequences, and hand-constructed encirclement scenarios produce exact expected flips.

## Architecture Notes

### Move Validation

`applyMove` delegates legality checks to existing Sprint 1 functions:
- PLACE → `placeCoin` (already throws on illegal placement)
- JOIN → `isLegalJoin` (already checks queen-line, blocking, duplicate, crossing)
- PASS → no validation required (caller is responsible for move selection)

This avoids duplicating legality logic and ensures `applyMove` is a thin orchestrator over existing validated primitives.

### Cycle Detection

When a JOIN connects two coins already in the same connected component, we find the shortest path between them in the existing edge graph (BFS), then concatenate the new edge to form the cycle boundary. The resulting `Position[]` is the closed polygon.

**Why shortest path**: In a planar graph with the no-crossing rule, the shortest path between two coins in the same component is the unique simple path that, together with the new edge, encloses the bounded region. The alternative (longer) path would wrap around the outside and not represent the interior.

**Complexity**: BFS on ≤ 12 vertices and ≤ 66 edges is trivially fast.

### Point-in-Polygon

Standard ray-casting algorithm on the planar embedding using the actual grid coordinates of cycle vertices. Coins at integer grid intersections are tested for strict interior containment.

**Edge case handling**: Coins exactly on cycle edges cannot exist (guaranteed by Sprint 1's no-blocking-coin rule), so we only need to handle strictly interior vs strictly exterior.

### Face Flips

For a simple JOIN: flip both endpoints.  
For a cyclic JOIN: flip both endpoints + all interior coins.  
For a PLACE: no flips (coin is placed with the specified face).  
For a PASS: no flips.

The flip operation is a pure function: `newCoins.set(key, { ...coin, face: flipFace(coin.face) })`.

## Testing Strategy

### Example Tests (move.test.ts)

- PLACE: empty board → coin appears, supply decreases, player switches
- Simple JOIN: edge added, endpoints flip, player switches
- PASS: passCount increments, player switches
- Cyclic JOIN rectangle: 4 boundary coins + 1 interior → interior flips
- Cyclic JOIN empty interior: only endpoints flip
- Illegal move rejection: occupied PLACE, blocked JOIN, duplicate JOIN

### Property Tests (oracle.test.ts update)

- Random legal move sequences from initial state → invariant checks at each step
- `coins.size + coinsRemaining === 12`
- Face parity: number of heads + tails = total coins (consistency check)
- `legalPlacements` and `legalJoins` remain valid after each move

### Geometry Property Tests (geometry.test.ts update)

- `pointInPolygon` matches brute-force grid enumeration for random polygons on the 7×7 grid
- Known shapes (square, triangle, star) produce expected interior/exterior results
