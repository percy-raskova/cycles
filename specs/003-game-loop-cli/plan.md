## Summary

Build a turn-manager and CLI harness on top of the existing CYCLES engine (`src/core/`). The turn manager (`GameSession`) will orchestrate alternating turns, detect forced passes, detect terminal game-over (two consecutive passes), and compute final scores. The CLI harness reads move commands from stdin, renders the board as ASCII art, and validates input. All new core logic remains pure TypeScript with zero React imports and 90% test coverage.

## Design Notes

### `src/core/session.ts`

**GameSession** holds the canonical `GameState` plus metadata:
- `history: readonly Move[]` — full move log for replay/debug
- `consecutivePasses: number` — tracks passes toward terminal detection
- `isTerminal: boolean` — game-over flag
- `winner: Player | "draw" | null` — computed after terminal

**Functions (pure)**:
- `createSession(firstPlayer?: Player): GameSession`
- `step(session: GameSession, move: Move): GameSession | { error: string }` — apply one move, return new session or validation error
- `hasLegalMoves(session: GameSession): boolean` — check if current player can PLACE or JOIN
- `computeFinalScore(session: GameSession): FinalScore` — count heads vs tails when terminal
- `getTurnPrompt(session: GameSession): string` — whose turn + coins remaining (testable string generation)

**Forced pass rule**: If `hasLegalMoves` is false, `step` auto-converts any input into a `PassMove` (or the caller can call a dedicated `forcePass` helper). Per the clarified spec, a voluntary `PASS` when legal moves exist is rejected as an error.

**Terminal detection**: After any pass (forced or otherwise), if `consecutivePasses >= 2`, set `isTerminal = true` and compute winner.

### `src/cli/parser.ts`

Parse stdin lines into `Move` objects:
- `P A1 H` → `PlaceMove` (coordinate system: A1 = row 0, col 0; G7 = row 6, col 6)
- `J A1 B2` → `JoinMove`
- `PASS` → `PassMove` (only valid when `hasLegalMoves` is false)
- Invalid format → `ParseError` with message

### `src/cli/renderer.ts`

ASCII board for 7×7 grid:
- Rows labeled 1–7, columns labeled A–G
- Empty intersections: `·` or `+`
- Coins: `H` (heads) or `T` (tails)
- Edges drawn between coins as `—`, `|`, `/`, `\`

The renderer is purely functional: `render(session: GameSession): string`.

### `src/cli/main.ts`

Entry point (`bun run cli` or `node dist/cli/main.js`):
1. Randomly select first player (HEADS or TAILS), print announcement
2. Loop until terminal:
   a. Print ASCII board
   b. Print turn prompt (`HEADS to move — 12 coins remain`)
   c. Read line from stdin
   d. Parse → `Move` or error
   e. Call `step(session, move)`
   f. If error, print and re-prompt (same player)
   g. If pass detected and no legal moves, print "Forced pass"
   h. Update session
   i. If terminal, print final score and winner, exit 0
3. On EOF (Ctrl+D), exit 0 gracefully

## Test Strategy

- **Unit tests** (`src/core/__tests__/session.test.ts`): GameSession state machine, forced pass, terminal detection, scoring, illegal move rejection, random first-player seeding
- **Property tests** (`src/core/__tests__/session.test.ts` with fast-check): Coin-count invariant across full sessions, legal-move count monotonicity, terminal never reverses
- **CLI unit tests** (`src/cli/__tests__/`): Parser round-trips, renderer output format, EOF handling
- **Integration-style test**: Simulate a full game via programmatic stdin/stdout to verify end-to-end flow

## Risk Notes

- **Random first player**: Must be deterministic in tests (seedable or mockable random source)
- **ASCII rendering complexity**: Drawing edges diagonally in monospace is tricky; MVP renderer may show coins and edges in simplified form (e.g., edge list below the board) if ASCII line drawing proves too complex for the sprint scope
- **Rule ambiguity discovery**: The spec explicitly calls this out. Any ambiguities found during terminal play must be documented in the spec and resolved before proceeding to the React UI
