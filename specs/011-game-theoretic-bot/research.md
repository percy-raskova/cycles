# Phase 0 Research: Game-Theoretic Bot

All decisions below resolve the NEEDS-CLARIFICATION items implied by the spec's
requirements and the performance budget. The spec's Session 2026-05-27
clarifications (Q1–Q5) are treated as settled inputs and are not re-litigated here.

---

## R1 — Score margin: σ = heads − tails, not own-face count

**Decision**: The Strategic bot computes its signed margin as
`sigma(state) = scoreForPlayer(state, "HEADS") − scoreForPlayer(state, "TAILS")`,
and converts to the side-to-move's perspective as `+sigma` when Heads, `−sigma`
when Tails. All Δσ algebra (FR-004, FR-005) is expressed against this σ.

**Rationale**: The existing `scoreForPlayer(state, player)` (`src/core/score.ts`)
returns *only the count of the player's own face* — not σ. The Greedy bot's
`after − before` therefore measures own-face gain, which silently disagrees with
the game-theoretic Δσ for non-cycle JOINs (a +2 Heads edge raises Heads count by 2
*and* lowers Tails count by 2, a swing of 4 in σ but only +2 in own-face count for
some configurations). The analysis document (§3) defines every move value on
σ = heads − tails, so the bot must too, or the heuristics will be miscalibrated.

**Alternatives considered**:
- *Reuse Greedy's own-face delta* — rejected: inconsistent with the Δσ algebra; would
  make cycle-close and JOIN heuristics wrong.
- *Add a `sigma()` to `src/core/score.ts` as shared API* — deferred: keep the new
  helper inside `strategic/sigma.ts` to avoid touching shared engine surface for a
  bot-specific concern. Promote later only if a second consumer appears (DRY without
  premature extraction).

## R2 — Search shape: 3-ply negamax with alpha–beta, deterministic move ordering

**Decision**: Implement a fixed depth-3 negamax (minimax from the side-to-move's
perspective; negamax = minimax for a zero-sum σ, noted inline in `search.ts` — resolves
analyze finding D1) with alpha–beta pruning. Children are generated by `allLegalMoves`
and ordered by the deterministic comparator (R6) for both pruning efficiency and
reproducibility.

**Leaf evaluation is STATE-based** (resolves analyze finding U1 / Clarification Q7):
`v(state) = W_SIGMA·σ_signed(state) + W_BOUNDARY·boundary(state)
+ W_CENTER_AVOID·centerExposure(state) + W_TEMPO·tempo(state)`, every term computed over
the *position*, side-to-move perspective. The exact signed σ is the dominant base term.
The non-cycle Δσ algebra and the cycle-close swing `2(t−h)` are **not** additive leaf
terms — they are realized by the search backup (applying a move changes the successor's
σ, which the leaf already measures). A cycle-closing move's value therefore equals
`Δσ_close + v(successor)` as an *emergent* property of the recursion, with no double
counting. The per-move Δσ/cycle-close evaluators (`deltaSigma`, `cycleClose`) are used
only for (a) move ordering / beam ranking and (b) the depth-0 static fallback ranking —
never re-summed into `v(state)`.

**Rationale**: Depth 3 is the spec mandate (Q2). Negamax collapses the
Heads-max / Tails-min cases into one sign-flipped routine, keeping per-function
complexity under the Biome ≤15 gate. Alpha–beta with good ordering reduces an
`O(b³)` full-width search toward `O(b^{1.5})` in the best case.

**Alternatives considered**:
- *Plain minimax (no pruning)* — rejected: cannot meet the time budget at early-game
  branching (see R3).
- *MCTS* (suggested in analysis §10.4) — rejected for v1: non-deterministic without a
  seeded RNG, harder to make reproducible, and overkill given the fixed shallow depth
  the spec requires. Recorded as a future enhancement.
- *Iterative deepening* — unnecessary at a fixed depth of 3; the timeout (R4) handles
  the worst case more simply.

## R3 — Meeting the time budget (SC-002: <100 ms avg, <2,000 ms worst)

**Problem**: Early-game branching is large: up to 49 empty positions × 2 faces ≈ 98
PLACE moves plus legal JOINs (often 0–10 early). At `b ≈ 100`, full-width depth-3 is
~10⁶ leaves, each calling `applyMove` (which clones a `Map`, appends an edge array,
and may run a BFS `findCycle` + point-in-polygon scan). That risks exceeding 100 ms.

**Decision**: Three mitigations, in order:
1. **Alpha–beta + heuristic move ordering** (R2/R6) — order PLACE children by the
   cheap static leaf heuristic (boundary/center) and JOIN children by static Δσ
   *before* recursing, so good moves prune early.
2. **Static pre-ranking with a search beam at interior nodes** (now sanctioned by
   Clarification Q6 — resolves analyze finding I1): at the root, score all candidates
   with the cheap static heuristic and recurse the *full* set (so FR-005 holds for the
   selected move). At interior nodes (depth ≥ 2), restrict expansion to the top-K
   children by static score (K = `K_BEAM`, default 12, in `weights.ts`; K = ∞ disables
   the beam). Alpha–beta stays the sound core; the beam is an explicit approximation
   layered on top. **Verification item (T024)**: confirm in `strategic-search.test.ts`
   + `bot-perf.test.ts` that K=12 keeps worst case <2,000 ms and does not regress the
   SC-001 win rate; widen/narrow K empirically (the Q3/Q6 tuning loop).
3. **Move-timeout cutoff** (R4) as the hard backstop.

**Rationale**: Symmetric placement faces mean most early PLACE positions are
near-equivalent; a static-score beam at deeper plies discards near-duplicates cheaply.
The root stays full-width so the *selected* move is always a fully-searched candidate.

**Alternatives considered**:
- *Full-width depth-3 with no beam* — rejected pending R3 verification; likely too slow
  in the opening. If perf tests show it fits, the beam can be disabled (K=∞) — decide
  empirically during implementation, do not assume.
- *Transposition table* — deferred: CYCLES positions rarely transpose within 3 plies
  (moves are mostly additive: PLACE grows coins, JOIN grows edges), so the table hit
  rate is low relative to its memory/hashing cost. Future enhancement.

## R4 — Timeout via injected clock; fallback behavior (Edge Cases, FR-008, Q8)

**Decision (injected-clock DI — resolves analyze finding N1 / Clarification Q8)**: The
core search NEVER references `performance`, `Date`, or any ambient clock — that would
make `src/core/` impure and non-deterministic, violating Constitution I & VI and FR-002.
Instead, `StrategicBotConfig` carries an **optional injected clock** `now?: () => number`
and `deadlineMs?: number`:

- **No clock injected (default)** — used by the exported `strategicBot`, tests,
  tournaments, and the CLI. The search runs to its fixed depth with zero time
  dependence. Output is fully pure and deterministic (FR-002). `src/core/` contains no
  time API.
- **Clock injected** — the UI (`src/ui/useBotGame.ts`, where timing is permitted) passes
  `now: () => performance.now()` and `deadlineMs: 2000`. The search records
  `deadlineAt = now() + deadlineMs` once at entry, then checks `now() >= deadlineAt` at
  interior nodes; on expiry it unwinds and returns the **best fully-evaluated root move
  so far**. If no root move completed (pathological), it returns `greedyBot(state)`.

Separately and independent of timing: if the state-based leaf evaluation yields an
all-zero score for every root candidate (no heuristic is non-neutral, FR-008), the bot
returns `greedyBot(state)`.

**Rationale**: DI keeps the engine pure (testable, platform-agnostic, deterministic)
while still giving production a hard worst-case bound (SC-002) and a guaranteed legal
move (SC-003). Greedy is already in-tree, pure, and fast — the natural fallback. The
2,000 ms default aligns with the existing UI bot delay in `useBotGame.ts`.

**Determinism test**: `strategic-search.test.ts` asserts that, with no clock injected,
repeated calls are byte-identical, and that injecting a *generous* clock (deadline never
hit) produces the same move as the no-clock path.

## R5 — Forced passes inside the search & terminal detection

**Decision**: The search treats a node as **terminal** when `state.passCount >= 2`
(matches `session.checkTerminal`). At a non-terminal node, if `allLegalMoves(state)`
is empty, the search applies a single forced `{ type: "PASS" }` via `applyMove` and
recurses *without decrementing depth* (a forced pass is not a real ply choice).
Terminal leaf value = `sigma(state)` from the side-to-move's perspective.

**Rationale**: The public `step()` auto-passes when no legal moves exist; the bot is
only *invoked* when it has moves, but its lookahead must model the opponent (or itself)
being forced to pass. Not decrementing depth on a forced pass prevents the search from
"wasting" its 3 plies on mandatory non-moves, keeping the strategic horizon intact.
Per spec 003, voluntary passing is forbidden, so PASS is only ever generated as a
forced transition — never as a candidate in `allLegalMoves`.

**Alternatives considered**:
- *Decrement depth on forced pass* — rejected: shortens effective lookahead exactly
  when the position is most constrained (late game), the opposite of what's wanted.

## R6 — Deterministic tie-breaking & move ordering (FR-002)

**Decision**: A single comparator orders moves for both (a) alpha–beta child ordering
and (b) final tie-break among equal total scores:
1. Primary: by total/static score, descending.
2. Tie-break by move-type rank: **cycle-closing JOIN < non-cycle JOIN < PLACE**
   (cycle-closing preferred first). Cycle-vs-non-cycle is determined by
   `findCycle(state, a, b) !== null`.
3. Within the same move type: by **lexicographically smallest position key**
   (`"row,col"`), using `positionKey`. For JOIN, compare the sorted `(a, b)` key pair;
   for PLACE, compare `position` then face (`heads` before `tails`).

**Rationale**: Exactly the rule in FR-002. `legalJoins` already emits position-sorted
pairs and `legalPlacements` iterates row-major, so the comparator is consistent with
existing enumeration order, minimizing surprise. A stable sort over a
fully-specified comparator guarantees identical output across runs and platforms
(determinism for SC-001/SC-003 reproducibility).

## R7 — Exhaustive endgame switch (Edge Cases)

**Decision**: Before launching depth-3 search, estimate the remaining game-tree leaf
count with a cheap product of branching factors over remaining plies. If the estimate
is ≤ 200, run **full exhaustive negamax to terminal** (no depth limit, no beam) and
play the game-theoretically optimal move; otherwise run depth-3. The estimator is
intentionally conservative (overestimates), so it only switches to exhaustive when
genuinely small, never risking a blow-up.

**Rationale**: Late positions (few empty positions, few non-intersecting JOINs left)
have tiny trees; exhaustive search there is both fast and *optimal*, directly serving
the "read the last 2–3 moves exhaustively" recommendation (analysis §11.5) and the
edge-case requirement. The 200-leaf threshold is a documented `weights.ts` constant,
tunable if perf tests show headroom.

**Alternatives considered**:
- *Always depth-3* — rejected: misses guaranteed-optimal endgame play that the spec
  explicitly calls for.
- *Counting exact tree size* — rejected: counting costs as much as searching; a cheap
  upper-bound estimate suffices to gate the switch.

## R8 — Initial heuristic weights (Q3: hand-tuned)

**Decision**: Seed `weights.ts` from the analysis confidence levels, then refine via the
Greedy tournament loop. Documented starting values (sign = good-for-side-to-move):

| Term | Const | Initial weight | Role | Source (analysis §) |
|------|-------|----------------|------|---------------------|
| Signed margin σ(state) | `W_SIGMA` | 1.0 | **leaf** (dominant base) | §2, §6 (exact) |
| Boundary/corner safety | `W_BOUNDARY` | 0.6 | **leaf** (positional) | §9a (90% conf) |
| Center exposure (rows/cols 2–4) | `W_CENTER_AVOID` | 0.4 | **leaf** (positional) | §9a (80% conf) |
| Tempo (no +2 either side, PLACE remains) | `W_TEMPO` | 0.3 | **leaf** (positional) | §9b/§9d (70% conf) |
| Non-cycle Δσ (+2/0/−2) | `W_DELTA_SIGMA` | 1.0 (identity on Δσ) | **ordering / static fallback** only | §3, §4 (95% conf) |
| Cycle-close swing `2(t−h)` | `W_CYCLE_CLOSE` | 1.0 (identity on Δσ_close) | **ordering / static fallback** only | §3, §9b (90% conf) |
| Interior-beam width | `K_BEAM` | 12 | R3 |
| Exhaustive-switch leaf threshold | `EXHAUSTIVE_LEAF_LIMIT` | 200 | R7 / spec |
| Default search depth | `SEARCH_DEPTH` | 3 | Q2 |
| Default deadline (ms) | `DEFAULT_DEADLINE_MS` | 2000 | SC-002 |

**Rationale**: Δσ-derived weights start at identity (the algebra is exact, highest
confidence); softer geometric heuristics get fractional weights proportional to their
stated confidence so the exact Δσ signal dominates ties. Final values are locked as
code constants once the 1,000-game tournament shows SC-001 (≥55% decisive win rate)
is met. **These are starting points, not final values** — the tuning loop (Q3) sets
the committed numbers.

**Alternatives considered**:
- *Auto-tuning / CMA-ES / self-play learning* — rejected for v1 (Q3 chose hand-tuning;
  also conflicts with the "no learning across games" assumption). Future enhancement.

## R9 — `inspectTopMoves(state, n)` design (Q5, FR-011)

**Decision**: Export `inspectTopMoves(state, n)` returning a readonly array of the
top-N candidates, each `{ move, totalScore, breakdown }` where `breakdown` is a
`Record<heuristicName, number>` of per-heuristic contributions (pre-weight raw score
and post-weight contribution both recorded). It reuses the same root evaluation the
bot uses, sorts via the R6 comparator, and **must not mutate `state`** (asserted by a
property test that deep-compares the input before/after).

**Rationale**: Directly satisfies FR-011 and SC-004 (developer inspectability) without
coupling to the search internals — it surfaces the per-heuristic leaf scores that drive
selection. Returning both raw and weighted contributions makes the hand-tuning loop
(R8/Q3) observable.

## R10 — UI integration & Greedy retention (FR-001, Assumptions)

**Decision**: `BotStrategyUI` becomes `"random" | "strategic"`; the SetupScreen replaces
the "Greedy" radio with "Strategic" (same a11y affordances, FR-009). `getBotFunction`
maps `"strategic" → strategicBot`. The async `yieldingGreedyBot` wrapper in
`useBotGame.ts` is replaced by a strategic equivalent only if perf testing shows the
synchronous bot blocks paint; otherwise the bot runs synchronously inside the existing
2,000 ms `setTimeout` (which already yields to the event loop). `greedyBot` and
`BotStrategy = "...|greedy"` remain in `src/core/bots/` for tournaments and fallback.

**Rationale**: FR-001 removes Greedy from the *selector* only, not the codebase
(Assumptions). Keeping the engine `BotStrategy` union inclusive of `"greedy"` preserves
`runSimulation` regression capability (FR-010, SC-001).

**Open verification item**: measure whether synchronous depth-3 search within the
2,000 ms timer causes a visible UI stall on a complex board. If yes, add a chunked/
async wrapper analogous to the existing `yieldingGreedyBot`; if no, drop the wrapper
entirely (simplest). Decide from the `bot-perf` measurement, do not assume.

## Implementation Corrections (discovered during `/speckit-implement`)

These supersede earlier decisions where noted; recorded here per Constitution VII
(document divergences) so the reasoning is preserved.

### IC-1 — σ swing of a non-cycle JOIN is ±4/0, not ±2/0 (corrects analysis §3)

`cycles-game-theory.md` §3 reports a non-cycle JOIN's `Δσ` as ±2; that quantity is
actually `Δ(#heads)`. The real `σ = #H − #T` swing is **double** (each flipped endpoint
moves σ by 2), i.e. **+4 / 0 / −4** (both-tails / mixed / both-heads, Heads perspective).
The engine (`applyMove`) is the source of truth (Constitution VII); the bot reasons in
real σ so the minimax leaf is the true win margin. `strategic-sigma.test.ts` cross-checks
the analytic Δσ against `applyMove` for every legal JOIN. Move-preference ordering is
unchanged, so heuristic direction is unaffected — only the magnitude/label is corrected.

### IC-2 — Minimax leaf must be antisymmetric; tempo moved to a root move-bias (revises Q7/R8)

Negamax requires a zero-sum leaf: `v(s,P) = −v(s,¬P)`. The σ, boundary-safety, and
center-exposure terms are antisymmetric, but **tempo is player-symmetric** (a property of
the position, "no +2 for either side"). Including it in the leaf broke antisymmetry and —
through the per-ply negations — biased the search so badly it preferred opening with the
*opponent's* face. Fix: the leaf is **σ + boundary + center only** (antisymmetric, sound),
and **tempo (FR-006) is applied once at the root** as a PLACE-over-JOIN bias
(`totalScore = backed-up value + W_TEMPO·tempo(move)`), never inside negamax. This is also
a more faithful reading of FR-006 ("defer JOINs"), which is a move preference, not a
position value.

### IC-3 — Interior beam replaced by adaptive full-width depth (revises Q6/R3)

A finite beam pruning the **opponent's response layer** is unsound in the placement phase:
many placements tie at 1-ply value, so the beam's tie-break kept tactically-irrelevant
cells and dropped the opponent's deep-best reply, making the search **overvalue
material-sacrificing lines** (verified: beam-off chose the correct opening, beam-on did
not). Replacement: **no beam**; instead adapt the full-width search depth to the branching
factor (`search.ts` `effectiveDepth`): full 3-ply when branching ≤ 16 (alpha-beta keeps
this ~b² and affordable), sound full-width 2-ply otherwise, and exhaustive for ≤200-leaf
endgames. This is both sounder and faster than the beam (measured avg ~19 ms / worst
~44 ms unloaded, vs the beam's ~73 ms). Move ordering now uses the accurate 1-ply value
`−v(successor)` (successors cached for reuse, so no extra `applyMove`), which also makes
alpha-beta prune well. `K_BEAM` defaults to ∞ (beam disabled); the knob remains for
experiments.

### IC-4 — SC-001 verified

With the corrected, sound bot, a 60-game Strategic-vs-Greedy calibration (both color
assignments, alternating starts) gave **Strategic 30 / Greedy 0 / 30 draws, 0 crashes** —
a 100% decisive win rate, far exceeding the SC-001 ≥55% target. The default weights (R8)
are therefore committed without further tuning; the gated 1000-game test asserts SC-001
on demand.
